## Goal

Close the four clinical-credibility gaps uncovered by the FoundationOne CDx head-to-head audit (`_workspace/board-polish/foundation_one_comparison.md`) so that BIKO GenomeBoard's AI Clinical Board can enter a supervised clinical pilot:

1. **No more LLM-invented drug/target pairings** (futibatinib mis-cited as a KRAS inhibitor) — introduce a *curate-then-narrate* architecture where OncoKB + CIViC variant-level evidence is resolved deterministically and the LLM is restricted to narrating the curated rows only.
2. **TP53 R249M must classify as Likely Pathogenic, not VUS** — the ACMG engine must recognise TP53 DNA-binding-domain hotspots (aa 175/245/248/249/273/282) as PM1-eligible even when VEP `DOMAINS` is empty or when ClinVar shows conflicting single-variant entries at that exact residue.
3. **Tier III VUS selector must stop admitting non-coding variants** (intronic/UTR/upstream) while unconditionally surfacing MMR-gene VUS (Lynch panel) regardless of hotspot status.
4. **Every therapy and biomarker claim must carry a PMID citation, and every report must show a patient-demographics header** — these are regulatory/credibility baselines that cannot be waived for a clinical pilot.

**Why this pattern.** LLMs hallucinate when asked to *generate* facts that exist in external databases. They do not hallucinate when asked to *paraphrase* a row they can see. The curate-then-narrate boundary (variants → deterministic resolver → LLM-narrator) turns treatment recommendations into a verifiable, testable pipeline artifact. Every line in the rendered Treatment Options table becomes traceable to a specific OncoKB or CIViC row, with a PMID.

### PM1 protein-domain hotspot (A3)

Current PM1 logic: fires only if VEP `DOMAINS` column is non-empty (`evidence_collector._has_domain`). This is **insufficient** for canonical cancer hotspots where:
- `DOMAINS` may be empty if the VCF's VEP run didn't request the domain field;
- the aa lies inside a narrow hotspot residue range that PM1 should recognise *by codon*, not by domain annotation;
- multiple ClinGen VCEP refinements (TP53, BRCA1/2, PTEN, MLH1, MSH2) define explicit codon-range PM1 triggers.

We introduce a new lookup table keyed by `(gene, residue_range, domain_name, source)`:

```json
{
  "TP53": [
    {"range": [175, 175], "domain": "DBD", "source": "Giacomelli 2018 PMID 30224644", "strength": "moderate"},
    {"range": [245, 249], "domain": "DBD L3 loop", "source": "Giacomelli 2018 + ClinGen SVI TP53 v2.0", "strength": "moderate"},
    {"range": [273, 273], "domain": "DBD", "source": "Giacomelli 2018 PMID 30224644", "strength": "moderate"},
    {"range": [282, 282], "domain": "DBD", "source": "Giacomelli 2018 PMID 30224644", "strength": "moderate"}
  ],
  "KRAS":  [ {"range": [12, 13], "domain": "P-loop (G-domain)", "source": "COSMIC v99 + OncoKB hotspots", "strength": "moderate"},
             {"range": [61, 61], "domain": "Switch II",           "source": "COSMIC v99 + OncoKB hotspots", "strength": "moderate"},
             {"range": [117, 117], "domain": "GTP-binding",       "source": "COSMIC v99",                   "strength": "supporting"} ],
  ...
}
```

PM1 fires when `gene` matches and `protein_position ∈ range`, even if `DOMAINS` is empty. Each entry carries an explicit PMID and a strength modifier (`moderate` = standard PM1, `supporting` = `PM1_Supporting`). Garrett 2021 PMID 33280026 (ClinGen SVI PM1 refinement) is the primary framework reference.

### ClinVar override reconciliation (A4)

Current behaviour: when ClinVar has a specific SCV for the exact variant with `Conflicting classifications of pathogenicity`, the engine treats that as VUS and does not let ACMG-engine evidence override it. This is what buries TP53 R249M as VUS.

Fix: when `(engine_class in {LP, P})` and `(clinvar_class == "Conflicting")` and `(PM1 fires from the protein-domain hotspot table)` and `(PM5 fires from an adjacent residue's P/LP ClinVar entry)`, the engine takes precedence and emits Likely Pathogenic with a `clinvar_override_reason` field. The reconciliation rule is encoded as a *narrow, auditable override*, not a blanket ClinVar ignore.

### Selector tightening (B1) + MMR carve-out (B2)

- B1: cancer-mode `_cancer_must_reason` and the Tier III VUS table rendered in `render.py` must reject any variant whose primary VEP consequence is not in `_PROTEIN_IMPACTING_CONSEQUENCES` (missense / frameshift / nonsense / splice-donor/acceptor / inframe indel / start-lost / stop-lost). Intronic, UTR, upstream, downstream, non_coding_transcript_*, and synonymous (unless splice-adjacent with SpliceAI ≥ 0.2) are rejected.
- B2: a new `_MMR_LYNCH_GENES = {"MLH1","MSH2","MSH6","PMS2","EPCAM"}` set. Cancer-mode MAY arm is extended: any protein-impacting VUS in an MMR gene is admitted as `VUS_MMR_Lynch` (priority 5.5 between `VUS_hotspot` and `VUS_TSG_LoF`). Rare-disease mode already surfaces these via HPO matching for Lynch syndrome queries, but the cancer-mode carve-out is new.

## Task B4 — Patient demographics header

**Clinical rationale.** F1 puts the patient block (name, DOB, sex, MRN, ordering physician, facility, specimen site/ID/type, collection/received dates) on every page. BIKO currently shows only Sample ID. A clinician reviewing a BIKO report cannot tell whose report they are holding — a regulatory non-starter for any CLIA-equivalent use.

**Layout approach (report-dev HIGH B4).** Implemented as a Jinja macro `{% macro patient_strip(patient, sample_id, date, lang) %}` in a new `templates/macros/patient.html`, mechanically replacing every `<div class="page-header-bar"></div>` in `templates/cancer/report.html` (and `templates/rare_disease/report.html` if present). WeasyPrint running-element approach (`@top-center`) is rejected — every `.page` div already carries `@page { margin: 0 }` and a manual strip, so the macro is the lowest-risk drop-in. Acceptance criterion: `grep -c "patient_strip(" templates/cancer/report.html` equals the number of `<div class="page">` occurrences.

**PHI gate (report-dev MED + db-dev REQ B4-db-1).** `reporting.persist_patient_metadata: false` gates **every** persistence sink, not just the KB SQLite. The full sink list (db-dev audit checklist):
1. `kb.sqlite3` — gene KB / variant cache / showcase cache
2. `_workspace/**/*.json` — any intermediate runner artifact
3. `docs/showcase/*.json` — the committed fixtures
4. `variant_cache.sqlite3` — cache-key namespace
5. `gene_knowledge.json` rebuild output
6. `scripts/db/version_manager.py` metadata (build provenance)

For every sink in the list, B4 adds a write-time guard: `if not config.get("reporting.persist_patient_metadata", False): patient = None` before serialization. A dedicated `test_phi_non_persistence.py::test_patient_name_never_reaches_kb_sqlite` (qa fixture 5 verbatim) scans every sink after an end-to-end run and asserts the patient name string is not present.

**Forward compat (db-dev REC B4-db-2).** The v2.3 report-DB will persist patient metadata in its own table with a separate retention policy. The v2.2 `reporting.persist_patient_metadata` flag is a temporary stopgap; the v2.3 design must supersede it with a per-case consent record. Documented here so the v2.3 planners do not re-implement ad-hoc gating.
